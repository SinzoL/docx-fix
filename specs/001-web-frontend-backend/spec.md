## Clarifications

### Session 2026-03-09

- Q: 用户上传文件时如何选择检查规则？ → A: 上传时同步选择检查模板（规则文件），已有的规则文件作为预置选项，另需配置一个"通用默认"规则作为兜底选项，默认只检查基础格式设置（页面设置、基本字体、基本段落格式等）。
- Q: 修复后的文件如何下载？ → A: 修复完成后先展示修复结果预览（修复前后对比），用户确认无误后点击"下载"按钮，通过浏览器下载功能直接触发文件下载（非跳转链接）。
- Q: 用户认证与访问控制？ → A: 选择方案 A——无需登录，匿名使用。每个浏览器会话自动分配临时 ID 用于隔离。
- Q: 用户上传的文件和生成的报告如何存储？ → A: 存储在用户本地浏览器中（IndexedDB/Cache API），使用长期缓存，默认有效期一个月，过期自动清理。
- Q: 前端组件库选择？ → A: TDesign React v1.12.0，搭配 TailwindCSS 3.4.17，用于数据展示和交互。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 上传文档并查看检查报告 (Priority: P1)

用户通过浏览器打开 Web 界面，上传一个 `.docx` 论文文件，并在上传区域同步选择检查模板（规则文件）。系统提供多个预置规则选项（如"哈工大(深圳)毕业论文中期报告"等已有规则）以及一个"通用默认检查"兜底选项。通用默认规则只检查基础格式设置（页面设置、基本字体、基本段落格式），适用于没有特定学校模板的用户。选择完成后系统运行格式检查，并在页面上以清晰、分类的方式展示检查报告，包括每一项的通过/警告/失败状态。

**Why this priority**: 这是整个 Web 界面的核心价值——将命令行的检查能力转化为可视化体验。没有上传和检查报告，其他功能无从谈起。

**Independent Test**: 上传一个已知有格式问题的 `.docx` 文件，分别使用"通用默认检查"和已有的学校规则进行检查，确认页面正确显示所有检查项及其状态（✓/⚠/✗），且不同规则下检查项不同。

**Acceptance Scenarios**:

1. **Given** 用户打开 Web 首页, **When** 用户选择一个 `.docx` 文件并选择检查模板（默认为"通用默认检查"）, **Then** 系统显示加载状态，使用所选规则完成检查后展示格式检查报告，每项标注 PASS/WARN/FAIL 及具体说明
2. **Given** 用户未主动选择检查模板, **When** 用户直接上传文件, **Then** 系统自动使用"通用默认检查"规则，只检查基础格式设置（页面设置、基本字体、基本段落格式）
3. **Given** 用户上传了一个非 `.docx` 文件（如 `.pdf`）, **When** 系统收到文件, **Then** 系统提示"仅支持 .docx 格式"并拒绝处理
4. **Given** 用户上传了一个完全合规的 `.docx` 文件, **When** 检查完成, **Then** 报告显示所有项均为 PASS，给出"格式合规"的总结
5. **Given** 用户上传了一个包含多项格式问题的文件, **When** 检查完成, **Then** 报告按类别（字体、编号、大纲级别等）分组显示问题，并标注每项是否可自动修复
6. **Given** 规则目录中有多个 YAML 规则文件, **When** 用户打开上传页面, **Then** 模板选择器列出所有可用规则（含名称和描述），"通用默认检查"排在第一位且为默认选中

---

### User Story 2 - 一键修复格式并下载 (Priority: P2)

用户在查看检查报告后，点击"一键修复"按钮。系统对可修复项执行自动修复，修复完成后展示修复结果预览页面，包括修复前后的检查状态对比（哪些项从 FAIL 变为 PASS）。用户在预览页面确认修复结果无误后，点击"下载修复文件"按钮，系统通过浏览器下载功能直接触发文件下载（而非跳转到新页面或提供外部链接）。

**Why this priority**: 修复是检查之后的自然延伸，也是用户最终想要达到的目标。检查报告让用户了解问题，修复让用户解决问题。预览环节让用户在下载前确认修复效果，增强信任感。

**Independent Test**: 上传一个有已知可修复问题的文件，点击修复按钮，确认预览页面正确展示修复前后对比，点击下载按钮后浏览器触发下载，下载的文件用 CLI `check` 检查，问题数量减少或为零。

**Acceptance Scenarios**:

1. **Given** 用户已上传文件且检查报告中有可修复项, **When** 用户点击"一键修复"按钮, **Then** 系统执行修复并显示修复结果预览页面，包含修复前后对比（如：FAIL→PASS 的项目列表）和"下载修复文件"按钮
2. **Given** 修复结果预览页面已展示, **When** 用户确认无误并点击"下载修复文件"按钮, **Then** 浏览器直接触发文件下载（通过 Blob + `<a>` click 或 Content-Disposition），文件名为原文件名加"_fixed"后缀
3. **Given** 用户已上传文件且检查报告中全部为 PASS, **When** 用户查看页面, **Then** "一键修复"按钮为禁用状态，提示"无需修复"
4. **Given** 修复过程中出现错误（如文件损坏）, **When** 系统检测到异常, **Then** 系统显示错误提示，不生成修复文件，不展示预览，原文件不受影响

---

### User Story 3 - 查看和切换规则集 (Priority: P3)

用户可以在上传页面的模板选择器中切换规则文件，也可以在检查报告页面切换规则后重新检查同一文件。用户可点击"查看规则详情"查看当前规则文件内容的可读化展示。系统自带一个"通用默认检查"规则，仅包含基础格式设置检查项（页面设置如纸张大小/页边距、基本字体如中文宋体/英文 Times New Roman、基本段落格式如行距/缩进），作为无特定学校模板时的兜底选项。

**Why this priority**: 规则驱动是项目核心原则之一，让用户直观理解和切换规则增强了工具的灵活性，但它不是最基本的使用流程。

**Independent Test**: 在报告页面切换到另一个规则文件，重新检查同一文件，确认检查结果与新规则一致。

**Acceptance Scenarios**:

1. **Given** 规则目录中有多个 YAML 规则文件, **When** 用户进入页面, **Then** 模板选择器列出所有可用规则（含名称和描述），"通用默认检查"排在第一位且为默认选中
2. **Given** 用户已上传文件并查看了检查报告, **When** 用户在报告页切换到另一个规则, **Then** 系统使用新规则重新检查同一文件并更新报告
3. **Given** 用户想了解规则详情, **When** 用户点击"查看规则详情", **Then** 以可读的列表形式展示规则文件中的各项检查规则及其参数
4. **Given** "通用默认检查"规则被选中, **When** 用户查看规则详情, **Then** 仅展示基础格式设置检查项（页面设置、基本字体、基本段落格式），无特定学校相关的检查项

### Edge Cases

- 上传文件过大（如超过 50MB）时，系统应提示文件大小限制并拒绝处理
- 用户在修复过程中刷新页面或关闭浏览器，系统应正确清理临时文件
- 多个用户同时上传文件，系统应隔离各用户的文件和处理过程
- 上传的 `.docx` 文件不含标准 OOXML 结构（如损坏的 ZIP 包），系统应优雅提示错误
- 规则目录中除了"通用默认检查"外无其他规则文件时，选择器仅显示默认选项
- 某个 YAML 规则文件格式有误（语法错误）时，系统应跳过该文件并在选择器中标注"不可用"
- "通用默认检查"规则文件不存在或被删除时，系统应使用硬编码的最小基础检查作为最终兜底
- 浏览器不支持 IndexedDB（如隐私模式下部分浏览器限制）时，系统仍可正常使用，仅不缓存历史记录，并提示用户
- 用户浏览器本地缓存空间不足时，系统应优雅降级，优先清理最旧的缓存记录

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供 Web 页面，允许用户通过浏览器上传 `.docx` 文件
- **FR-002**: 系统 MUST 在上传区域同步提供检查模板（规则文件）选择器，列出所有可用规则及其名称和描述
- **FR-003**: 系统 MUST 提供一个"通用默认检查"规则作为兜底选项，该规则仅检查基础格式设置：页面设置（纸张大小、页边距）、基本字体（中文字体、英文字体、字号）、基本段落格式（行距、缩进、对齐）
- **FR-004**: 系统 MUST 将"通用默认检查"设为默认选中项，排在选择列表第一位
- **FR-005**: 系统 MUST 调用现有的 `checker.py` 检查引擎，使用用户所选的规则文件对上传文件进行格式检查
- **FR-006**: 系统 MUST 以结构化、分类的方式在页面展示检查报告（包括 PASS/WARN/FAIL 状态）
- **FR-007**: 系统 MUST 提供"一键修复"功能，调用现有的 `fixer.py` 修复引擎处理可修复项
- **FR-008**: 系统 MUST 在修复完成后展示修复结果预览页面，包含修复前后的检查状态对比（FAIL→PASS 的项目列表），让用户确认修复效果
- **FR-009**: 系统 MUST 在修复预览页面提供"下载修复文件"按钮，点击后通过浏览器下载功能直接触发文件下载（Blob + `<a>` click 或 Content-Disposition），而非跳转链接
- **FR-010**: 系统 MUST 支持用户在检查报告页面切换规则后重新检查同一文件
- **FR-011**: 系统 MUST 展示当前规则的详细内容（可读化格式）
- **FR-012**: 系统 MUST 拒绝非 `.docx` 格式的文件上传，并给出友好提示
- **FR-013**: 系统 MUST 对上传文件大小施加限制（默认 50MB），超限时给出提示
- **FR-014**: 系统 MUST 无需登录即可使用，为每个浏览器会话自动分配临时 ID 用于隔离不同用户的文件和处理过程
- **FR-015**: 系统 MUST 在处理过程中显示进度/加载状态
- **FR-016**: 系统 MUST 将用户上传的文件和生成的检查报告缓存在浏览器本地存储中（IndexedDB / Cache API），实现长期缓存，用户再次访问时可查看历史记录
- **FR-017**: 系统 MUST 自动扫描 `rules/` 目录，将所有 YAML 规则文件（通过 `meta.name` 和 `meta.description` 字段）作为可选模板呈现给用户
- **FR-018**: 系统 MUST 对浏览器本地缓存设置默认有效期为 1 个月，过期数据自动清理
- **FR-019**: 系统 SHOULD 在用户再次访问时展示历史检查记录列表（来自本地缓存），方便用户快速回顾之前的检查结果

### Key Entities

- **文档（Document）**: 用户上传的 `.docx` 文件，包含文件名、大小、上传时间、处理状态
- **检查报告（Check Report）**: 对某个文档运行检查后的结果集合，包含多个检查项及其状态，绑定所使用的规则集
- **检查项（Check Item）**: 单条检查结果，包含规则名称、状态（PASS/WARN/FAIL）、详细说明、是否可修复
- **修复结果（Fix Result）**: 修复操作的结果，包含修复前后的状态对比、修复后文件的 Blob 数据
- **规则集（Rule Set）**: `rules/` 目录下的 YAML 规则文件，包含 `meta.name`、`meta.description`、检查/修复的规则定义；分为"预置规则"（如学校模板规则）和"通用默认规则"
- **通用默认规则（Default Rule Set）**: 系统内置的兜底规则，仅包含基础格式设置检查：页面设置（A4 纸张、标准页边距）、基本字体（中文宋体/英文 Times New Roman、常用字号）、基本段落格式（行距、首行缩进、对齐方式）
- **本地缓存（Local Cache）**: 浏览器 IndexedDB/Cache API 中存储的用户历史数据，包含上传文件、检查报告、修复文件，有效期 1 个月，过期自动清理

## Testing Strategy *(mandatory)*

### 测试原则

1. **每个新特性必须编写对应的测试**：每完成一个任务（Task），MUST 同步编写该任务涉及的测试用例
2. **全量回归测试**：每完成一个特性后，MUST 运行完整的测试套件（前端 + 后端），确保所有已有测试全部通过，不引入回归问题
3. **测试先于集成**：在将新特性标记为完成之前，MUST 确保其测试通过 AND 全量回归测试通过

### 后端测试 (pytest)

- **框架**: pytest + httpx（AsyncClient 测试 FastAPI）
- **测试目录**: `backend/tests/`
- **运行命令**: `cd backend && python -m pytest tests/ -v`
- **覆盖范围**:
  - `tests/test_rules_service.py` — 规则扫描服务单元测试（YAML 解析、默认规则排序、语法错误处理）
  - `tests/test_checker_service.py` — 检查引擎封装单元测试（调用 checker.py、结果序列化）
  - `tests/test_fixer_service.py` — 修复引擎封装单元测试（调用 fixer.py、修复前后对比）
  - `tests/test_api_rules.py` — GET /api/rules API 集成测试
  - `tests/test_api_check.py` — POST /api/check API 集成测试（正常检查、文件类型校验、大小校验、规则校验）
  - `tests/test_api_fix.py` — POST /api/fix + GET /api/fix/download API 集成测试
  - `tests/test_api_rules_detail.py` — GET /api/rules/{rule_id} API 集成测试
  - `tests/test_edge_cases.py` — 边缘情况测试（损坏文件、规则缺失、并发 session 等）
- **测试数据**: `backend/tests/fixtures/` 目录存放测试用 `.docx` 文件和 YAML 规则文件

### 前端测试 (Vitest)

- **框架**: Vitest + @testing-library/react + jsdom
- **测试目录**: `frontend/src/__tests__/`
- **运行命令**: `cd frontend && npx vitest run`
- **覆盖范围**:
  - `__tests__/services/api.test.ts` — API 调用封装测试（mock fetch）
  - `__tests__/services/cache.test.ts` — IndexedDB 缓存管理测试（使用 fake-indexeddb）
  - `__tests__/components/UploadPanel.test.tsx` — 上传面板组件测试（文件选择、规则加载、上传触发）
  - `__tests__/components/CheckReport.test.tsx` — 检查报告组件测试（状态展示、分类分组、修复按钮状态）
  - `__tests__/components/FixPreview.test.tsx` — 修复预览组件测试（前后对比展示、下载触发）
  - `__tests__/components/RuleDetail.test.tsx` — 规则详情组件测试
  - `__tests__/components/HistoryList.test.tsx` — 历史记录组件测试
  - `__tests__/App.test.tsx` — 应用状态机流转集成测试

### 回归测试流程

每完成一个 Phase 或 User Story 后，MUST 执行以下回归测试流程：

```bash
# 1. 运行后端全量测试
cd backend && python -m pytest tests/ -v

# 2. 运行前端全量测试
cd frontend && npx vitest run

# 3. 确认所有测试通过后，才进入下一个任务
```

**回归测试失败处理**：如果任何已有测试失败，MUST 先修复回归问题再继续新功能开发。回归修复优先级高于新功能开发。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从打开页面到看到检查报告的全流程可在 30 秒内完成（普通大小的论文文件）
- **SC-002**: 修复后的文件重新检查，可修复项的通过率达到 100%
- **SC-003**: 界面展示的检查报告与 CLI `check` 命令输出的结果完全一致
- **SC-004**: 用户可在 4 次点击内完成"上传 → 检查 → 修复 → 预览 → 下载"全流程
- **SC-005**: 系统在连续上传 10 个文件后仍保持正常响应，无内存泄漏或临时文件堆积
- **SC-006**: 90% 的目标用户（熟悉论文写作的大学生/研究生）无需说明即可完成首次使用

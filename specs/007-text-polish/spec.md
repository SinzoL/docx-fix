# Feature Specification: 内容润色 — LLM 驱动的学术文本表达优化

**Feature Branch**: `007-text-polish`  
**Created**: 2026-03-10  
**Status**: Draft  
**Input**: 用户希望在格式检查/修复之外，增加"内容润色"功能——利用 LLM 对文档中的文本表达进行优化（语病修正、用词润色、句式优化、学术规范化），同时确保原始语义不变、格式不被破坏。

## 背景与动机

### 现有系统的能力边界

当前 docx-fix 系统提供三层能力：

| 层级 | 说明 | 代表模块 | 状态 |
|------|------|---------|------|
| **L1 格式检查/修复** | 对比 XML 属性与 YAML 规则，修复格式属性 | `checker/`、`fixer.py` | ✅ 已实现 |
| **L2 文本排版习惯** | 检测文本层排版问题（标点、空格、全半角）| `text_convention_checker.py` | ✅ 已实现 |
| **L3 内容润色** | 优化文本表达质量（语病、用词、句式） | — | ❌ **待实现** |

L1 和 L2 都不读取文本的"语义"——L1 只看格式属性，L2 只做模式匹配。而 L3 是一个本质不同的功能：它需要**理解文本含义**，然后在**保持语义不变**的前提下优化表达。

### 用户痛点

学术论文写作中，常见的文本表达问题包括：

1. **语病**：语法错误、搭配不当、成分残缺、语序不当（如"这个实验表明了该方法具有较好效果"→ 多余的"了"）
2. **口语化**：学术论文中使用口语化表达（如"我们用了一种很好的方法"→ 应改为"本文采用了一种有效的方法"）
3. **用词重复**：同一段落中反复使用相同词汇
4. **标点不规范**：中英文标点混用、引号不规范（已由 L2 部分覆盖，但 L3 可进一步优化）
5. **句式问题**：过长句子难以理解、被动句过多、逻辑连接不清晰

这些问题需要**自然语言理解能力**，正则匹配无法有效解决，必须借助 LLM。

### 核心设计原则

> **"只改表达，不改内容"**

内容润色区别于格式修复（"只改格式，不改内容"）和文本排版（"只改字符模式，不改语义"），它的边界是：

- ✅ **改善表达方式**：语病修正、口语→学术、句式优化
- ❌ **不改变论述内容**：论点、论据、结论、数据、引用
- ❌ **不改变文档格式**：字体、字号、行距、缩进等格式属性

### Constitution 影响

> **Principle II（只改格式，不动内容）的 Justified Extension**：
>
> 内容润色修改的是文本字符串（与 L2 文本排版修复类似），这是对 Principle II 的进一步扩展。但与 L2 不同，L3 的修改涉及**语义层面的表达优化**（如同义词替换、句式重组），风险更高。
>
> **缓解措施**：
> - 润色功能完全独立于格式检查/修复流程，用户需主动选择进入润色模式
> - 所有润色建议逐条展示 Diff 对比，用户必须逐条接受/拒绝
> - 引入**双 Agent 审核机制**：polisher 润色后由 reviewer 审核语义一致性
> - 润色前自动备份原文件
> - 润色功能不依赖 YAML 规则（不是规则驱动，而是 LLM 驱动）

> **Principle VI（简洁优先）的合规性**：
>
> 新增的 `polisher/` 子包遵循现有 `checker/` 子包的模块化架构，不引入新的核心依赖（复用现有的 `openai` SDK 和 `python-docx`）。前端新增独立的 Tab 组件，不修改现有检查/修复流程。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 段落级文本润色：LLM 驱动的学术表达优化 (Priority: P1)

用户上传一份 Word 文档，进入"内容润色"模式。系统提取文档中的可润色段落（跳过目录、图注、表注、公式、参考文献列表等），分批发送给 LLM 进行表达优化。

LLM 对每个段落返回：润色后的文本、修改类型（语病/用词/标点/句式/学术规范）、修改说明。如果段落表达已经合适，不做修改。

润色过程通过 SSE 流式返回，前端渐进式渲染，用户可以在润色进行中实时看到已完成的润色建议。

**Why this priority**: 段落级润色是整个功能的核心能力，没有它其他用户故事无法运作。

**Independent Test**: 上传一个包含"这个实验表明了该方法具有较好效果"的文档，系统应返回润色建议"该实验结果表明此方法具有较好的效果"，类型为"语病修正"。

**Acceptance Scenarios**:

1. **Given** 段落文本为 "这个实验表明了该方法具有较好效果", **When** 润色引擎处理该段落, **Then** 返回润色建议，change_type 为 "grammar"，explanation 说明去除了多余的"了"
2. **Given** 段落文本为 "我们使用了一种很好的方法来解决这个问题", **When** 润色引擎处理该段落, **Then** 返回润色建议将"我们"改为"本文"、"很好的"改为"有效的"等，change_type 为 "wording"
3. **Given** 段落文本已经是规范的学术表达, **When** 润色引擎处理该段落, **Then** 返回 modified=false，不生成润色建议
4. **Given** 段落包含数据 "精度达到 95.3%", **When** 润色引擎处理该段落, **Then** 数据数值不被修改
5. **Given** 段落包含专有名词 "TensorFlow" 和 "ResNet-50", **When** 润色引擎处理该段落, **Then** 专有名词不被修改
6. **Given** 段落为目录（TOC）段落, **When** 系统提取可润色段落, **Then** 该段落被跳过
7. **Given** 段落为 "图3-1 实验结果对比" 的图注, **When** 系统提取可润色段落, **Then** 该段落被跳过
8. **Given** 文档有 50 个可润色段落, **When** 润色引擎分批处理（每批 5 段）, **Then** 10 批依次处理，每批完成后通过 SSE 推送给前端
9. **Given** 某段落前后各有 2 段上下文, **When** 润色引擎处理该段落, **Then** 上下文段落作为 Prompt 上下文发送（但只润色目标段落）

### User Story 2 — 润色预览与逐条审阅：Diff 对比 + 接受/拒绝 (Priority: P1)

润色完成后，前端展示完整的润色预览页面。每条润色建议以 Diff 对比形式展示（原文 vs 润色后），用户可以：

- **逐条接受/拒绝**：点击 ✅ 接受或 ❌ 拒绝单条建议
- **全部接受 / 全部拒绝**：批量操作
- **按类型筛选**：只看语病/用词/句式等某一类修改
- **查看修改说明**：每条建议附带 LLM 的修改理由

用户确认后，点击"应用选中的修改并下载"，系统仅将用户接受的修改写回文档，保留原始格式。

**Why this priority**: 逐条审阅是保证"不改变原意"的关键机制，用户必须有完全控制权。

**Independent Test**: 润色报告中有 10 条建议，用户接受其中 6 条、拒绝 4 条，下载的文档应仅包含 6 条修改。

**Acceptance Scenarios**:

1. **Given** 润色报告包含 10 条建议, **When** 用户在预览页面查看, **Then** 10 条建议按段落顺序排列，每条显示 Diff 对比、修改类型标签和修改说明
2. **Given** 用户点击某条建议的 ❌ 拒绝按钮, **When** 用户随后点击"应用并下载", **Then** 该条修改不被应用到文档中
3. **Given** 用户点击"全部接受", **When** 所有建议变为接受状态, **Then** 用户仍可单独拒绝某条
4. **Given** 用户选择"按类型筛选 → 语病修正", **When** 列表更新, **Then** 只显示 change_type 为 "grammar" 的建议
5. **Given** 某条建议的修改说明为 "学术论文中避免使用'我们'，改用'本文'", **When** 用户展开详情, **Then** 能看到完整的修改说明文本
6. **Given** 润色报告汇总统计, **When** 用户查看, **Then** 顶部显示"共 10 条建议：语病 3 | 用词 4 | 句式 2 | 标点 1"

### User Story 3 — 格式保留回写：精确修改 Run 文本，保留原格式 (Priority: P1)

当用户确认润色修改后，系统将修改写回 Word 文档。回写过程必须**精确保留原段落的所有格式属性**（字体、字号、加粗、斜体、颜色等），仅修改 Run 的文本内容。

回写策略：
- 如果修改完全在某个 Run 内部（如 "非常重要" → "至关重要"），直接修改该 Run 的 text
- 如果修改跨越 Run 边界，合并受影响的 Runs 并保留第一个 Run 的格式
- 回写后自动备份原文件（`.polish.bak` 后缀）

**Why this priority**: 格式保留是内容润色区别于简单文本替换的关键，如果回写破坏了格式，功能就失去了价值。

**Independent Test**: 一个段落有 3 个 Runs（宋体 + 加粗宋体 + 宋体），润色修改了加粗 Run 中的文字，回写后加粗格式应保留。

**Acceptance Scenarios**:

1. **Given** 段落 `[宋体:"这是一个"][宋体加粗:"非常重要"][宋体:"的实验"]`, 润色文本为 "这是一个至关重要的实验", **When** 回写引擎执行, **Then** `[宋体:"这是一个"][宋体加粗:"至关重要"][宋体:"的实验"]`，加粗格式保留
2. **Given** 某 Run 的字体为 Times New Roman 12pt, **When** 润色修改了该 Run 的文本, **Then** 回写后字体仍为 Times New Roman 12pt
3. **Given** 用户只接受了 3 条修改（第 5、12、30 段）, **When** 回写引擎执行, **Then** 仅修改这 3 个段落，其他段落完全不变
4. **Given** 回写前, **When** 系统执行回写操作, **Then** 自动创建 `xxx.polish.bak` 备份文件
5. **Given** 润色建议将 "使用了Python语言" 改为 "采用了Python语言", 且 "Python" 横跨两个 Run, **When** 回写引擎执行, **Then** 正确处理跨 Run 修改，保留 "Python" 的格式

### User Story 4 — 双 Agent 语义守护：polisher + reviewer 审核 (Priority: P2)

为防止 LLM 在润色过程中意外改变原文语义，系统采用双 Agent 机制：

1. **Polisher Agent**: 负责润色文本（优化表达）
2. **Reviewer Agent**: 负责审核润色结果（检查语义一致性）

Reviewer Agent 接收原文和润色后文本，判断是否存在语义偏移。如果检测到语义变化，该条润色建议会被标记为 "⚠️ 语义可能有变化"，提醒用户重点审阅。

**Why this priority**: 双 Agent 是质量保障机制，增加了 LLM 调用成本但显著降低了语义偏移风险。作为 P2 是因为即使没有 reviewer，用户也可以通过逐条审阅来把关。

**Independent Test**: 如果 polisher 将"该算法的时间复杂度为 O(n²)"改为"该算法的时间复杂度较高"（丢失了具体复杂度信息），reviewer 应标记为语义偏移。

**Acceptance Scenarios**:

1. **Given** polisher 将 "实验在 GPU 环境下进行" 改为 "实验在高性能计算环境下进行", **When** reviewer 审核, **Then** 判断为语义可能有变化（从具体"GPU"变为笼统"高性能计算"），标记 ⚠️
2. **Given** polisher 将 "我们使用了" 改为 "本文采用了", **When** reviewer 审核, **Then** 判断为语义不变（仅主语代词替换），不标记
3. **Given** reviewer 审核失败（LLM 调用异常）, **When** 系统处理, **Then** 该条建议不标记 reviewer 结果，用户仍可正常审阅和接受/拒绝
4. **Given** 双 Agent 审核开启, **When** 润色处理文档, **Then** 每条建议的处理流程为：polisher 润色 → reviewer 审核 → 合并结果

### User Story 5 — 前端润色入口：独立 Tab + 状态管理 (Priority: P1)

在 Web 界面的上传面板区域，新增"内容润色"Tab 入口。用户可以选择：
- "格式检查"（现有功能）
- "内容润色"（新功能）

选择"内容润色"后，上传文档即进入润色流程，不触发格式检查。润色完成后进入预览模式，整体交互流程独立于格式检查流程。

**Why this priority**: 前端入口是用户接触润色功能的第一步，必须清晰直观。

**Independent Test**: 点击"内容润色"Tab，上传文档后，系统进入润色流程而非检查流程。

**Acceptance Scenarios**:

1. **Given** 用户打开 Web 界面, **When** 查看上传面板, **Then** 看到两个 Tab："格式检查"（默认选中）和"内容润色"
2. **Given** 用户切换到"内容润色"Tab, **When** 上传 docx 文件, **Then** 系统调用润色 API 而非检查 API
3. **Given** 润色处理中, **When** 用户查看页面, **Then** 显示进度条和已完成的润色建议（渐进式渲染）
4. **Given** 润色完成, **When** 用户查看页面, **Then** 自动进入润色预览模式（Diff 对比视图）
5. **Given** 用户在润色模式, **When** 切换回"格式检查"Tab, **Then** 润色状态被保留，切回时仍在

### Edge Cases

- 空文档（无文本段落）：应提示"文档无可润色的文本内容"
- 文档仅包含表格/图片、无正文段落：提取可润色段落为空，提示用户
- 段落文本为纯英文：跳过中文语病检查，仅做英文表达优化（如果后续支持）
- 段落文本 < 5 个字符：跳过润色（太短无意义）
- 段落文本 > 2000 个字符：单独作为一批处理（不与其他段落合并）
- LLM 返回的润色文本与原文完全相同：视为无修改，不生成建议
- LLM 返回格式异常（非 JSON）：重试 1 次，仍失败则跳过该批段落并记录 warning
- 网络中断或 LLM API 不可用：已完成的润色建议正常展示，未完成的标记为"润色中断"
- 回写时文档被锁定（只读）：提示用户
- 段落包含 OMath（公式）元素：跳过该段落不润色
- 段落包含超链接：润色时保留超链接 Run 不修改
- 表格单元格中的文本：首期不支持润色，仅润色主体段落（body paragraphs）

## Requirements *(mandatory)*

### Functional Requirements

#### 段落提取与分类

- **FR-001**: 系统 MUST 提取文档主体段落（`doc.paragraphs`）的文本用于润色
- **FR-002**: 系统 MUST 跳过以下不可润色段落：TOC 段落、图注（"图X-Y"开头）、表注（"表X-Y"开头）、纯公式段落（包含 OMath）、参考文献列表项、页码/页眉/页脚
- **FR-003**: 系统 MUST 跳过文本长度 < 5 字符的段落
- **FR-004**: 系统 MUST 记录每个段落的 Run 结构信息（文本 + 格式），用于后续精确回写
- **FR-005**: 首期 MUST 仅处理主体段落（body paragraphs），表格单元格和脚注暂不支持

#### LLM 润色引擎

- **FR-006**: 系统 MUST 分批将段落发送给 LLM 进行润色，每批 5-8 段
- **FR-007**: 每批段落 MUST 附带前后各 1-2 段作为上下文（仅润色目标段落）
- **FR-008**: LLM Prompt MUST 明确约束不得改变原文的论点、论据、结论、数据、公式、引用编号、专有名词
- **FR-009**: 每条润色建议 MUST 包含：段落索引、原始文本、润色后文本、修改类型（grammar/wording/punctuation/structure/academic）、修改说明、置信度
- **FR-010**: 如果段落表达已合适，LLM MUST 返回 modified=false，不生成建议
- **FR-011**: LLM 调用 MUST 有重试机制（最多 2 次重试），失败时跳过该批次并记录 warning
- **FR-012**: 润色过程 MUST 通过 SSE 流式返回（每完成一批推送一次）

#### 双 Agent 语义守护

- **FR-013**: 系统 SHOULD 支持 Reviewer Agent 审核润色结果的语义一致性
- **FR-014**: Reviewer 检测到语义偏移时，MUST 在建议中标记 `semantic_warning: true`
- **FR-015**: Reviewer Agent 为可选功能（可通过配置关闭），默认开启

#### 格式保留回写

- **FR-016**: 回写 MUST 精确保留每个 Run 的格式属性（字体、字号、加粗、斜体、颜色、上下标等）
- **FR-017**: Run 内部修改 MUST 直接修改 Run.text，不改变 Run 结构
- **FR-018**: 跨 Run 修改 MUST 合并受影响的 Runs 并保留第一个 Run 的格式
- **FR-019**: 回写前 MUST 自动备份原文件（`.polish.bak` 后缀）
- **FR-020**: 系统 MUST 仅回写用户接受的润色建议

#### 前端交互

- **FR-021**: 上传面板 MUST 新增"内容润色"Tab，与"格式检查"Tab 并列
- **FR-022**: 润色预览 MUST 以 Diff 对比形式逐条展示每条建议
- **FR-023**: 用户 MUST 能逐条接受/拒绝润色建议
- **FR-024**: 用户 MUST 能一键全部接受/全部拒绝
- **FR-025**: 用户 MUST 能按修改类型筛选建议
- **FR-026**: 润色处理中 MUST 显示进度指示和已完成的建议（渐进式渲染）
- **FR-027**: 每条建议 MUST 显示修改类型标签、修改说明和 reviewer 标记（如有）

#### API 接口

- **FR-028**: MUST 提供 `POST /api/polish` 接口（上传文档+执行润色，SSE 流式返回润色报告）
- **FR-029**: MUST 提供 `POST /api/polish/apply` 接口（应用用户选中的修改，生成修改后文档）
- **FR-030**: MUST 提供 `GET /api/polish/download/{session_id}` 接口（下载润色后文档）

### Key Entities

- **ParagraphSnapshot（段落快照）**: 记录段落的文本、位置、样式、Run 结构信息，用于提取和回写
- **PolishSuggestion（润色建议）**: 单条润色建议，包含原文、润色后文本、修改类型、说明、reviewer 标记
- **PolishReport（润色报告）**: 所有润色建议的集合 + 统计信息
- **PolishApplyRequest（应用请求）**: 用户选中要接受的建议索引列表

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 润色建议中语义偏移率 < 5%（通过 reviewer agent 和人工抽查验证）
- **SC-002**: 回写后文档的格式属性与原文档 100% 一致（通过自动化对比验证）
- **SC-003**: 50 段落文档的润色处理时间 < 60 秒（含 LLM 调用时间）
- **SC-004**: 用户可以在润色进行中实时看到已完成的建议（SSE 流式延迟 < 2s）
- **SC-005**: 所有现有测试套件通过，润色功能不影响格式检查/修复流程

## Testing Strategy *(mandatory)*

### 后端单元测试 (pytest)

- `test_text_extractor.py` — 段落提取测试：
  - 正常段落正确提取
  - TOC/图注/表注/公式段落被跳过
  - 短文本（< 5 字符）段落被跳过
  - Run 结构信息完整记录
  - 段落分批逻辑（每批 5-8 段）

- `test_polish_engine.py` — 润色引擎测试（mock LLM）：
  - 正常润色返回正确的 PolishSuggestion
  - 无需修改的段落返回 modified=false
  - LLM 返回异常格式时重试
  - 重试失败时跳过并记录 warning
  - 上下文窗口正确附带前后段落
  - reviewer agent 正确标记语义偏移

- `test_text_writer.py` — 格式保留回写测试：
  - Run 内部修改保留格式
  - 跨 Run 修改保留第一个 Run 格式
  - 仅回写用户接受的建议
  - 未修改的段落完全不变
  - 备份文件自动创建

- `test_diff_calculator.py` — Diff 计算测试：
  - 字级别差异正确计算
  - 无差异返回空列表
  - 中英文混合文本正确处理

### 前端测试 (Vitest)

- `PolishPanel.test.tsx` — Tab 切换和上传测试
- `PolishPreview.test.tsx` — Diff 对比展示和接受/拒绝交互测试

### 回归测试

每完成一个 Task 后，MUST 运行完整的测试套件确保不引入回归问题。

## 约束与风险

| 约束/风险 | 详细说明 | 对策 |
|-----------|---------|------|
| **LLM 改变原意** | LLM 可能在"优化表达"时引入语义偏移 | 双 Agent（polisher + reviewer）+ 用户逐条审阅 |
| **回写破坏格式** | Run 合并或文本替换可能影响格式属性 | ParagraphSnapshot 记录完整 Run 结构；回写前后格式校验 |
| **Token 消耗大** | 50 段文档约需 10 批 LLM 调用 + reviewer 调用 | 分批处理 + 跳过不需要润色的段落 + 缓存已润色段落 |
| **长文档超时** | 100+ 段文档润色可能超过 60s | SSE 流式返回渐进式渲染；前端可在润色进行中查看已完成建议 |
| **中英文混排** | 学术论文常有中英混排，LLM 需正确处理 | Prompt 明确英文术语/专有名词不修改 |
| **依赖 LLM 可用性** | 润色功能完全依赖 LLM API | LLM 不可用时提示用户，不影响格式检查/修复功能 |
| **Constitution Principle II** | 内容润色修改文本内容，是对 Principle II 的 justified extension | 独立于格式修复流程；用户主动选择；逐条审阅；自动备份 |
